第三章 · 数据库设计与搭建（数据层）
================================================================

  数据库是项目的"地基"。地基打不好，房子建再高也会塌。

【3.1 什么是数据库】

  数据库就是"结构化存储数据的地方"。类比 Excel：
  - 数据库（Database）= 一个 Excel 文件
  - 表（Table）= 一个 Sheet
  - 行（Row）= 一行数据
  - 列（Column）= 一列字段

  本项目使用 PostgreSQL（关系型数据库），存储 8 张表。

【3.2 什么是 ORM（TypeORM）】

  你不用直接写 SQL（数据库语言），而是写 TypeScript 类（Entity）。
  TypeORM 自动把类的属性翻译成数据库的列。

  比如：
    @Entity('products')
    export class Product {
      @Column() name: string;  // ← 这一行自动变成数据库的"name"列
    }

  类比：TypeORM 就像一个"即时翻译官"。
  你对它说中文（TypeScript），它自动翻译成日语（SQL）告诉数据库。
  数据库用日语回答，它再翻译回中文告诉你。

【3.3 项目中的 8 张表（Entity）】

  ① products 表 — 商品目录
  ┌──────────┬──────────┬──────────────────────┐
  │ 字段       │ 类型       │ 说明                   │
  ├──────────┼──────────┼──────────────────────┤
  │ id         │ TEXT(PK)   │ 商品唯一标识，如 shoe-001│
  │ name       │ TEXT       │ 商品名称，如 "Nike Air Max 90 黑白"│
  │ category   │ TEXT       │ 类目，如"运动鞋"        │
  │ brand      │ TEXT       │ 品牌，如"Nike"          │
  │ attributes │ JSONB      │ 扩展属性 {color:"黑白", style:"休闲"}│
  │ imageUrl   │ TEXT       │ 商品图片网络地址         │
  └──────────┴──────────┴──────────────────────┘

  ② price_history 表 — 历史价格（每个商品在每个平台每个月的价格）
  ┌──────────┬──────────────┬──────────────────────┐
  │ 字段       │ 类型           │ 说明                   │
  ├──────────┼──────────────┼──────────────────────┤
  │ id         │ SERIAL(PK)    │ 自增主键               │
  │ productId  │ TEXT          │ 关联商品 ID             │
  │ platform   │ TEXT          │ 平台名（淘宝/京东/拼多多）│
  │ price      │ DECIMAL(10,2) │ 价格（精确到分）         │
  │ date       │ DATE          │ 日期（如"2025-01-01"）   │
  └──────────┴──────────────┴──────────────────────┘
  种子数据量：30 商品 × 3 平台 × 7 个月 = 630 条

  ③ users 表 — 用户账户
  phone / passwordHash(bcrypt加密) / nickname / wechatOpenid 等

  ④ review_summaries 表 — 评价摘要
  productId / platform / positiveKeywords[] / negativeKeywords[] / summary

  ⑤ favorite_lists 表 — 收藏清单
  userId / name / isPublic

  ⑥ favorite_items 表 — 收藏商品
  listId / productId

  ⑦ search_history 表 — 搜索历史
  userId / imageUrl / resultSnapshot(JSONB) / createdAt

  ⑧ price_alerts 表 — 降价提醒
  userId / productId / targetPrice / platform / isActive

【3.4 怎么连接数据库】

  database.config.ts（数据库连接配置文件）：
    export const databaseConfig = {
      type: 'postgres',            // 数据库品牌
      host: 'localhost',           // 地址（本机）
      port: 5432,                  // 端口号（PostgreSQL 默认）
      username: 'postgres',        // 用户名
      password: '298556',          // 密码
      database: 'smart_shopping',  // 数据库名
      entities: [Product, PriceHistory, ...],  // 要管理的表
      synchronize: true,           // 自动建表（开发环境用）
    };

  synchronize: true 意味着：
  你改了 Entity（比如加了一个字段），重启服务后数据库自动更新表结构。
  类比：就像"自动装修公司"，你说要加一个插座（字段），
  公司在你不在家的时候自动施工完成。

  生产环境会改成 synchronize: false，手工管理数据库变更（Migration）。

第四章 · 服务端模块开发（业务逻辑层）
================================================================

  服务端的核心设计思想是"模块化"。
  每个模块是一个功能单元，有明确的输入（谁可以调）和输出（返回什么）。
  模块之间尽量独立，通过接口（API）通信。

【4.0 模块开发通用步骤】

  类比：新建一个模块就像在公司里"成立一个新部门"。
  部门需要三个核心角色：
  1. Controller（前台接待员）：接待外部请求，分发任务
  2. Service（技术人员）：实际干活的人，处理业务逻辑
  3. Module（部门注册信息）：告诉公司（AppModule）这个部门存在

  开发步骤：
  Step 1: 创建文件夹 server/src/modules/<模块名>/
  Step 2: 写 xxx.service.ts（业务逻辑，含数据库操作）
  Step 3: 写 xxx.controller.ts（定义 API 端点，@Get/@Post 等）
  Step 4: 写 xxx.module.ts（注册 Controller + Service + 数据库实体）
  Step 5: 在 app.module.ts 的 imports 中加入 XxxModule

【4.1 认证模块（AuthModule）— 项目的"门禁系统"】

  API 端点：
  ┌──────────────────────────────────────────────────────┐
  │ POST /auth/register    { phone, code, password }     │
  │   注册新用户。验证短信验证码 → bcrypt 加密密码        │
  │   → 返回 JWT 双令牌 + 用户信息                        │
  │                                                      │
  │ POST /auth/login       { phone, password }           │
  │   用户登录。查询手机号 → bcrypt 比对密码              │
  │   → 返回 JWT 双令牌 + 用户信息                        │
  │                                                      │
  │ POST /auth/refresh     { refreshToken }              │
  │   刷新令牌。验证 RefreshToken → 签发新 AccessToken    │
  │                                                      │
  │ POST /auth/send-sms-code { phone }                  │
  │   发送短信验证码（Demo 固定返回 123456）               │
  └──────────────────────────────────────────────────────┘

  JWT 令牌 — 不是传统"会话"
  传统方式：服务器记住"谁登录了"（Session），存在内存里。
  重启服务器 → 所有人都要重新登录。

  JWT 方式：服务器不记"谁登录了"。它"签发一张签名门票"。
  门票里有用户 ID + 有效期 + 数字签名。
  用户可以拿这张门票访问服务器，服务器"验票"确认身份。
  签名防伪：门票是加密签名的，用户无法伪造。

  签名过程（sign）：把 { userId, phone } + 密钥 → 签名 → Token
  验签过程（verify）：拿出 Token → 用密钥验证签名 → 读用户信息

  类比：JWT Sign = 景区售票处给出"盖章门票"。
  JWT Verify = 景区入口查票员"验章"。
  验章通过即放行，不需要打电话回售票处确认。

  为什么用双令牌：
  AccessToken（15分钟有效期）：用于日常验证。即使被盗，15分钟后失效。
  RefreshToken（7天有效期）：只存在手机安全存储中，仅用于刷新 AccessToken。
  这样即使网络窃听，攻击者也只能拿到短暂的 AccessToken。

【4.2 图像识别模块（RecognitionModule）— 项目的"核心引擎"】

  API：POST /recognition/upload { images: ["base64...", ...] }

  工作流程（数据在代码中的流转路径）：
  照片 Base64 → DTO 校验（@IsArray/@ArrayMinSize 确保至少 1 张图）
  → RecognitionController.upload()
  → RecognitionService.recognize()
    ├─ 有 AI Key → 调豆包多模态 API（OpenAI 兼容格式）
    │   请求体：{ model, messages: [{ role: "system", content: "你是商品识别助手..." }, { role: "user", content: [image_url, text] }] }
    │   解析返回的 JSON 提取 category/brand/attributes
    └─ 无 AI Key → Mock 返回 { category: "运动鞋", brand: "Nike" }
  → matchProducts(category, brand)
    查询 products 表 WHERE category = '运动鞋'
    → 对每个匹配商品，子查询各平台最新价格：
      SELECT ... FROM price_history WHERE date = (SELECT MAX(date) FROM price_history WHERE same productId AND same platform)
    → 计算最低价 → 附 shopType 映射 { 淘宝:C店, 京东:旗舰店, 拼多多:旗舰店 }
  → 返回 { category, brand, attributes, products[] }
  → 自动保存搜索历史（从请求头 Authorization 提取 JWT → 解析 userId）

  图片怎么传输：Base64 编码
  原始照片是二进制数据（0 和 1 组成），无法直接在 JSON 文本中传输。
  Base64 是一种编码方式，把二进制数据转成纯文本的字母数字串。
  服务端收到后可以还原为图片。

  类比：Base64 就像"把一盒巧克力（二进制）装在快递盒（JSON）里邮寄"。
  巧克力直接寄会碎，装盒子里就安全了。收件人打开盒就能还原。

【4.3 商品/比价/历史模块 — 数据查询三件套】

  ① ProductModule（商品搜索）
  API：GET /products/search?category=运动鞋&brand=Nike
  使用 PostgreSQL 的 ILIKE（大小写不敏感模糊匹配）：
  WHERE category ILIKE '%运动鞋%' AND brand ILIKE '%Nike%'
  每个匹配商品附最新各平台价格。

  为什么用 ILIKE 而不是 LIKE：
  LIKE 区分大小写（'nike' 找不到 'Nike'），ILIKE 不区分。
  用户输入"nike"或"NIKE"都能找到。

  ② ComparisonModule（单商品比价）
  API：GET /comparison/:productId
  聚合一个商品在各平台的最新价格，计算最低价和最低价平台。

  ③ HistoryModule（历史价格趋势）
  API：GET /products/:id/history
  响应结构变化（2.2 版本后新增 lowestPrice 字段）：
  {
    productId,                              // 商品 ID
    lowestPrice: { price: 749, platform: "拼多多", date: "2024-06-01" },
    platforms: { "淘宝": [{date, price}...], "京东": [...], ... }
  }
  遍历所有价格记录，按 platform 分组，同时找出全时段最低价。

【4.4 NLP 模块 — 自然语言→结构化筛选】

  API：POST /nlp/filter { query, conversationId?, context? }

  工作流程：
  query "要便宜点的Nike旗舰店"
  → 有 AI Key：调用豆包 API
    System Prompt：定义输出 Schema
    Few-shot 示例：教模型理解"便宜点→price_asc"
    返回 JSON { sort: "price_asc", shopType: "旗舰店", brand: "Nike" }
  → 无 AI Key：applyRules() 关键词规则引擎
    13 条规则：
    /便宜|低价|实惠|省钱|最便宜|最低价/ → sort = price_asc
    /贵|高价|最贵|最高价/               → sort = price_desc
    /旗舰店/                            → shopType = 旗舰店
    /自营/                              → shopType = 自营
    /C店|淘宝店|个人店/                 → shopType = C店
    品牌关键词（Nike/耐克/Adidas/阿迪达斯）
    /运动鞋|跑步鞋/                     → category = 运动鞋
    X以下/以内/不超过                    → priceRange.max = X
    X以上/起/起步                        → priceRange.min = X
    X到Y / X至Y                         → priceRange = { min:X, max:Y }
  → validateFilter() 白名单校验
    sort: 仅允许 default / price_asc / price_desc
    shopType: 仅允许 旗舰店 / C店 / 自营
    priceRange: min ≥ 0, max ≥ 0，非法值丢弃
    未知字段: 静默丢弃，不抛异常

  多轮对话机制：
  每次请求返回 conversationId（UUID 随机生成），
  客户端下次请求带上同一 conversationId → 后端可 Redis 缓存对话上下文。
  Demo 阶段未接入 Redis，conversationId 仅透传。

  类比：NLP 模块就像"餐厅的传菜翻译"。
  顾客说"便宜点、大份的"（自然语言），
  翻译成后厨的标准化指令（结构化 JSON）：
  "排序方式=价格升序，分量=大份"。

【4.5 收藏/搜索历史/评价/降价提醒模块 — Demo 增强功能】

【4.6 安全基础设施（Common 层）】

  四道防线，层层设防：

  防线1 — Helmet HTTP 安全头：
  自动设置 10+ 个 HTTP 安全头（XSS 防护、点击劫持防护等）。
  一行代码：app.use(helmet())

  防线2 — Throttler 限流：
  全局 100 请求/分钟。超过限制返回 429 Too Many Requests。
  类比：就像银行窗口"每人每次限办一项业务"，防止黄牛占窗口。

  防线3 — JWT 守卫（JwtAuthGuard）：
  标记 @UseGuards(JwtAuthGuard) 的端点必须带有效 Token。
  未带 Token → 自动返回 401 Unauthorized。

  防线4 — ValidationPipe 全局校验：
  所有请求数据必须通过 DTO 规则校验。
  规则写在 DTO 类中，使用 class-validator 装饰器：
    @IsString()  → 必须是字符串
    @IsNotEmpty()→ 不能为空
    @IsArray()   → 必须是数组
    @Matches(/^1[3-9]\d{9}$/) → 必须符合手机号格式

  重要教训（踩过的坑）：
  ValidationPipe 配置了 whitelist: true（白名单模式），
  只保留 DTO 中声明了验证装饰器的字段。
  如果 DTO 字段没有加 @IsString() 等装饰器，
  该字段会被静默删除，导致数据丢失。
  → 本项目所有 DTO 字段均已加装饰器。

【5.8 Dio 拦截器（api_service.dart）— 网络层核心】

  Dio 实例创建时注册一个 InterceptorsWrapper，包含两个钩子：

  onRequest（请求发出前）：
  await _storage.getAccessToken()
  → 有 Token：options.headers['Authorization'] = 'Bearer $token'
  → 无 Token：跳过（公开接口不需要认证）

  onError（收到错误响应后）：
  if (error.response?.statusCode == 401) {
    // 尝试自动刷新 Token
    读取 RefreshToken → POST /auth/refresh → 拿到新 AccessToken
    → 保存新 Token → 重试原请求
    → 失败 → clearTokens() → 页面应跳转 /login
  }

  类比：这个拦截器就像一个"智能助手"。
  每次出门前自动帮你带门卡（onRequest），
  门卡过期时自动去前台换新卡（onError），
  换卡也失败才叫你自己去登记（跳登录页）。

第六章 · Docker 容器化部署
================================================================

【什么是 Docker】

  Docker 让你的应用在任何机器上"开箱即用"。
  不需要手动安装 PostgreSQL、配置环境变量、解决版本冲突。
  一个命令启动所有服务。

  docker-compose.yml 文件描述了"需要启动哪些服务"：

  ┌──────────────┬──────────────────────────────────┐
  │ 服务           │ 配置                               │
  ├──────────────┼──────────────────────────────────┤
  │ postgres      │ postgres:16-alpine 镜像           │
  │               │ 环境变量：用户名/密码/数据库名      │
  │               │ 端口映射：5432:5432                │
  │               │ 数据卷：pgdata（数据持久化）        │
  │               │ 健康检查：pg_isready 每 5 秒        │
  ├──────────────┼──────────────────────────────────┤
  │ redis         │ redis:7-alpine 镜像               │
  │               │ 端口映射：6379:6379                │
  │               │ 数据卷：redisdata（AOF 持久化）     │
  │               │ 健康检查：redis-cli ping           │
  ├──────────────┼──────────────────────────────────┤
  │ app           │ 构建上下文：./server                │
  │               │ 端口映射：3000:3000                │
  │               │ 依赖：postgres + redis（等健康检查）│
  │               │ 环境文件：.env.docker               │
  └──────────────┴──────────────────────────────────┘

  启动命令：docker-compose up -d     （-d = 后台运行）
  停止命令：docker-compose down
  查看日志：docker-compose logs -f app

  docker-compose.yml 中的 depends_on + healthcheck：
  app 容器不会在 postgres 和 redis 就绪前启动。
  避免"数据库还没启动，App 就连数据库报错"的竞态问题。

  Volumes（数据卷）的作用：
  容器删除后，数据卷保留。
  类比：容器是"租的房子"，数据卷是"自己的家具"。
  退租（删容器）时家具（数据）不会被房东扔掉。

  .env vs .env.docker：
  .env          → 本地开发用（DB_HOST=localhost, REDIS_HOST=localhost）
  .env.docker   → Docker 用（DB_HOST=postgres, REDIS_HOST=redis）
  Docker 内部用"服务名"作为主机名，Docker DNS 自动解析。

第八章 · 技术术语速查表（按字母序）
================================================================

  ┌─────────────────┬───────────────────────────────────────────────┐
  │ 术语              │ 解释 + 类比                                     │
  ├─────────────────┼───────────────────────────────────────────────┤
  │ API              │ Application Programming Interface            │
  │                  │ 应用程序接口。两个软件之间通信的"契约"。       │
  │                  │ 类比：餐厅里的菜单。                              │
  ├─────────────────┼───────────────────────────────────────────────┤
  │ AsyncNotifier    │ Riverpod 的一种 Provider，管理异步数据。        │
  │                  │ 类比：外卖配送员。状态=等待接单/取餐中/已送达。  │
  ├─────────────────┼───────────────────────────────────────────────┤
  │ Base64           │ 二进制数据→纯文本的编码方式。                  │
  │                  │ 用于在 JSON 中传输图片。                       │
  │                  │ 类比：把照片放进信封（文本格式）邮寄。           │
  ├─────────────────┼───────────────────────────────────────────────┤
  │ bcrypt           │ 密码哈希算法。不可逆（不能从哈希反推密码）。     │
  │                  │ 类比：碎纸机。一旦碎掉，无法还原。              │
  ├─────────────────┼───────────────────────────────────────────────┤
  │ CORS             │ Cross-Origin Resource Sharing                │
  │                  │ 允许不同来源的请求（如手机访问电脑服务器）。      │
  │                  │ 类比：小区门禁白名单。                          │
  ├─────────────────┼───────────────────────────────────────────────┤
  │ CRUD             │ Create/Read/Update/Delete，数据四操作。        │
